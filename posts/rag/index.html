What is Retrieval-Augmented Generation (RAG)?

In which we approximate RAG using only LLMs and no vector database (for the Roman Empire's Wikipedia page)

This is the 43rd post in my series 50 Days of LIT Prompts.

If you're only here to ask an AI about the Roman Empire, be my guest, but consider reading the following first.

If you've been with us since the beginning, you know I think most folks are looking through the wrong end of the telescope when it comes to using large language models (LLMs). That is, too many folks want to feed in 5 words and get out 500 (e.g., write an essay for me). LLMs are sentence completion machines, searching only to find the next most plausible set of words. The larger the prompt you give them the more constrained their output. Conversely, sparse prompts seeking large outputs are asking for trouble. It's like setting off a rocket without any fins. Things might start off "okay," but small changes in direction can lead to problems very quickly. Sometimes this is what you want. It can be good to have in improv partner, but a lot of the time, folks like to avoid hallucinations. So, I tend to say more folks should be feeding in 500 words and expecting 5. That's a wordy way of saing these tools are pretty good at summariztion and entity extraction. I wouldn't, however, trust LLMs to produce answers any different than an average internet user, and we need to remember that the average internet user and LLMs are the products of society at large, biases et al. included.

So what are we to do about the dream of an AI-powered FAQ? Can we find a way to use LLMs to help answer question the average internet user wouldn't know? We'll as we observed back in that first post, " If you want an LLM, or the man on the street, to summarize a text, have them read it first." That is, you can turn things around and look through the right end of the telescope by making the "article" you want to question AND your question part of the same prompt. We've seen this work with short articles and webpages. However, as you may have noticed, there is a limit to how much information we can place in a prompt. Also, how are we supposed to include both a question and the answer to that question in the same prompt. If we have the answer, why are we asking the question? Enter retrieval-augmented generation (RAG), a phrase you may have heard some "AI folks" throwing around.

Use your question to perform a search over a set of text in which you expect the answer will be found and prepend the "trusted text" of your search results to your prompt, sandwiching some instructions between it and your question. That is, you augment your text generation by retrieving some set of text to help build a better prompt and get a better answer. The search does most of the heavy lifting. Instead of a vector database, we'll use an LLM. As you'll see below I've taken the full text of this Wikipedia page on the Roman Empire, placing it into a set of prompts, each coresponding to a section of the page (i.e., "History", "Geography and demography", "Languages", "Society", "Government and military", "Economy", "Architecture and engineering", "Daily life", "Arts", "Literacy, books, and education", "Religion", "Legacy").

“The output of an LLM is a secondary source, and just like any secondary source, it doesn't remove the need to check primary sources.”

The first template takes in a question and asks an LLM to return the section most likely to contain the answer. The LLM's matchs are then used to retrieve a section of the page which we prepend to the prompt, just like we described above. The result is a system that can field individual questions of the Wikipedia page. It's not perfect; if it's hard to place your question in one of the above categories, it will error. However, it does let us do things we couldn't otherwise. Mostly, the model we're using couldn't do this if we tried to feed in the whole page because it's too big. By breaking things up, we've found a way to look through the right end of the telescope. Does this make LLM's 100% reliable? No, and don't believe anyone who tells you otherwise. The output of an LLM is a secondary source, and just like any secondary source, it doesn't remove the need to check primary sources. That being said, secondary sources, like Wikipedia, exist for a reason. They're a useful, and often more accessable, starting place.

LIT Prompts is a browser extension built at Suffolk University Law School's Legal Innovation and Technology Lab to help folks explore the use of Large Language Models (LLMs) and prompt engineering. LLMs are sentence completion machines, and prompts are the text upon which they build. Feed an LLM a prompt, and it will return a plausible-sounding follow-up (e.g., "Four score and seven..." might return "years ago our fathers brought forth..."). LIT Prompts lets users create and save prompt templates based on data from an active browser window (e.g., selected text or the whole text of a webpage) along with text from a user. Below we'll walk through a specific example.

The Prompt Patterns (Templates)

Maps are models; they don't show everything. That's okay as long as you don't confuse the map for the territory.

When crafting a LIT Prompts template, we use a mix of plain language and variable placeholders. Specifically, you can use double curly brackets to encase predefined variables. If the text between the brackets matches one of our predefined variable names, that section of text will be replaced with the variable's value. Today we'll be using {{passThrough["info"]}}. See the extension's documentation.

If the text within brackets is not the name of a predefined variable, like {{What do you want to know about the Roman Empire?}}, it will trigger a prompt for your user that echo's the placeholder (e.g., a text bubble containing, "What do you want to know about the Roman Empire?"). After the user answers, their reply will replace this placeholder. A list of predefined variables can be found in the extension's documentation.

To use these templates, trigger the first one and ask away.

Ask about the Roman Empire

In a moment I will provide you with a question about the Roman Empire. Your job is to match the question to the category most likely to contain the answer. Here's a list of categories: 

categories = ["History", "Geography and demography", "Languages", "Society", "Government and military", "Economy", "Architecture and engineering", "Daily life", "Arts", "Literacy, books, and education", "Religion", "Legacy"]

---

Provide your answer in the form of a JSON object with two key-value pairs. The first key-value pair has the key "question" and a value equal to the above question. If the question contains double quotes escape them like so `\\"`. The second key-value pair had the key is "next" and the value of the category most likely to contain the questions answer.

And here are the template's parameters:

  • Output Type: LLM. This choice means that we'll "run" the template through an LLM (i.e., this will ping an LLM and return a result). Alternatively, we could have chosen "Prompt," in which case the extension would return the text of the completed template.
  • Model: gpt-4o-mini. This input specifies what model we should use when running the prompt. Available models differ based on your API provider. See e.g., OpenAI's list of models.
  • Temperature: 0. Temperature runs from 0 to 1 and specifies how "random" the answer should be. Since we're seeking fidelity to a text, I went with the least "creative" setting—0.
  • Max Tokens: 500. This number specifies how long the reply can be. Tokens are chunks of text the model uses to do its thing. They don't quite match up with words but are close. 1 token is something like 3/4 of a word. Smaller token limits run faster.
  • JSON: Yes. This asks the model to output its answer in something called JSON, which is a nice machine-readable way to structure data. See https://en.wikipedia.org/wiki/JSON
  • Output To: Hidden. We can output the first reply from the LLM to a number of places, the screen, the clipboard... Here, I've chosen the hide the output entirely. This is uesful when passing output to another template.
  • Post-run Behavior: DYNAMIC. Like the choice of output, we can decide what to do after a template runs. Here we will trigger the promt named in the passThrough["next"] variable.
  • Hide Button: unchecked. This determines if a button is displayed for this template in the extension's popup window.

  • On the Dangers of Stochastic Parrots: Can Language Models Be Too Big? 🦜 by Emily M. Bender, Timnit Gebru, Angelina McMillan-Major, and Shmargaret Shmitchell. There's a lot of history behind this paper. It was part of a chain of events that forced Timnit Gebru to leave Google where she was the co-lead of their ethical AI team, but more than that, it's one of the foundational papers in AI ethics, not to be confused with the field of "AI safety," which we will discuss later. It discusses several risks associated with large language models, including environmental/financial costs, biased language, lack of cultural nuance, misdirection of research, and potential for misinformation. If you want to engage critically with LLMs, this paper is a must read.
  • Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks by Patrick Lewis, Ethan Perez, Aleksandra Piktus, Fabio Petroni, Vladimir Karpukhin, Naman Goyal, Heinrich Küttler, Mike Lewis, Wen-tau Yih, Tim Rocktäschel, Sebastian Riedel, and Douwe Kiela. This is where we get the name "Retrieval-Augmented Generation." From the abstract:
    Large pre-trained language models have been shown to store factual knowledge in their parameters, and achieve state-of-the-art results when fine-tuned on downstream NLP tasks. However, their ability to access and precisely manipulate knowledge is still limited, and hence on knowledge-intensive tasks, their performance lags behind task-specific architectures. Additionally, providing provenance for their decisions and updating their world knowledge remain open research problems. Pre-trained models with a differentiable access mechanism to explicit non-parametric memory can overcome this issue, but have so far been only investigated for extractive downstream tasks. We explore a general-purpose fine-tuning recipe for retrieval-augmented generation (RAG) -- models which combine pre-trained parametric and non-parametric memory for language generation. We introduce RAG models where the parametric memory is a pre-trained seq2seq model and the non-parametric memory is a dense vector index of Wikipedia, accessed with a pre-trained neural retriever. We compare two RAG formulations, one which conditions on the same retrieved passages across the whole generated sequence, the other can use different passages per token. We fine-tune and evaluate our models on a wide range of knowledge-intensive NLP tasks and set the state-of-the-art on three open domain QA tasks, outperforming parametric seq2seq models and task-specific retrieve-and-extract architectures. For language generation tasks, we find that RAG models generate more specific, diverse and factual language than a state-of-the-art parametric-only seq2seq baseline.
